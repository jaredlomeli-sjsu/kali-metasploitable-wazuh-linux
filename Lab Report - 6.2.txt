1. Objective
The purpose of this lab was to explore Wazuh's built-in File Integrity Monitoring (FIM) module, which watches designated directories for unauthorized file changes by comparing cryptographic hashes over time. This builds directly on the manual hashing work from Lab 3.1, only here the hashing and comparison happen automatically and continuously inside a SIEM rather than being run by hand. SEC+ treats FIM as a core control for detecting tampering, malware drops, and unauthorized configuration changes, so this exercise connected that exam topic to a hands-on demonstration of how a real SOC tool implements it.
2. Lab Environment
This lab used the full three-VM lab topology. Kali Linux ran the Wazuh agent and hosted the monitored /etc directory used for the test file. The Wazuh Manager VM (Ubuntu Server) ran the Manager, Indexer, and Dashboard components installed in Part 8 of the guide, and received the FIM events generated by the agent. Metasploitable 2 was left suspended for the duration of this lab, per the RAM guidance in Part 8, since it was not needed for any FIM step. All three VMs sit on the same Host-Only network, with the Wazuh Dashboard viewed from a browser on the Windows host over HTTPS.
Kali Linux: ran the Wazuh agent and held the monitored test file.
Wazuh Manager VM: received, analyzed, and stored the FIM events; hosted the Dashboard.

3. Step-by-Step Procedure
I started by checking the syscheck section of the Wazuh agent's configuration file on Kali, to confirm which directories were already being watched by default.
I created a new test file inside one of those monitored directories, /etc, and wrote a line of placeholder content into it.
I waited a few minutes to give Wazuh's syscheck process time to complete a scan and record a baseline hash for the new file.
I then overwrote the file's content to simulate an unauthorized change to a system file.
I opened the Wazuh Dashboard in a browser on the Windows host and navigated to the Integrity Monitoring module to look for alerts referencing the test file.
I reviewed the alert details, which showed both a file added event and a file modified event, each with their associated SHA-256 hash values.
Finally, I removed the test file from /etc to leave the monitored directory clean.
4. Commands & Actions Performed
sudo grep -A2 "syscheck" /var/ossec/etc/ossec.conf
This searches the Wazuh agent's configuration file for the syscheck block and prints two lines of context after the match, showing which directories (such as /etc, /usr/bin, and /usr/sbin) are configured for integrity monitoring by default.
sudo touch /etc/testfile_wazuh.txt
This creates an empty test file inside /etc, a directory Wazuh already monitors, so the new file will be picked up by the next syscheck scan.
echo "original content" | sudo tee /etc/testfile_wazuh.txt
This writes an initial line of text into the test file using tee, which lets the write happen with sudo privileges while still working inside a pipe.
echo "content was changed!" | sudo tee /etc/testfile_wazuh.txt
This overwrites the test file's content, simulating an unauthorized modification that Wazuh's FIM module should detect on its next scan.
sudo rm /etc/testfile_wazuh.txt
This deletes the test file once the demonstration is complete, returning the monitored directory to its original state.

6. Results & Observations
The Integrity Monitoring module in the Dashboard showed two distinct events for the test file: a file added event when it was first created, and a file modified event after the content was changed, exactly as expected from the lab instructions.
Each event included the SHA-256 hash of the file both before and after the change, and the two hash values had no resemblance to each other despite the file content differing by only a few words. This is the same avalanche effect observed manually in Lab 3.1, now triggered automatically by the SIEM rather than by running sha256sum by hand.
Because /etc is one of the directories Wazuh monitors by default, no extra configuration was needed to pick up the test file. This confirmed that the default FIM configuration is broad enough to catch changes to common system locations without any tuning.
7. What I Learned
This lab made the practical value of FIM concrete by tying it back to the hashing concepts from Lab 3.1. Manually running sha256sum on a file is useful for understanding the math behind hashing, but it does not scale, no analyst is going to hash every file on a server by hand every few minutes. Wazuh's syscheck module automates exactly that process, continuously baselining and re-checking monitored files and raising an alert the moment a hash changes.
For SEC+, this reinforced that FIM is one of the practical controls that protects the Integrity leg of the CIA Triad, and that it is especially relevant to detecting malware drops, unauthorized configuration changes, and tampering with critical system files. It also reinforced why default monitored paths matter: because /etc is already covered out of the box, a real attacker modifying a configuration file there would generate an alert without any extra setup, which is the kind of out-of-the-box detection coverage SEC+ expects security professionals to understand and rely on.
