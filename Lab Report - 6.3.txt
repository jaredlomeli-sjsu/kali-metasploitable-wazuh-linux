1. Objective
The purpose of this lab was to navigate the Wazuh Dashboard the way a SOC analyst would during a shift, rather than focusing on a single attack or detection technique. The goal was to build familiarity with where different categories of security data live inside a SIEM interface, which is a practical skill on its own and also reinforces the SEC+ Domain 4 concepts around SIEM, alert triage, and log correlation covered throughout the earlier labs in this section.
2. Lab Environment
This lab used the Wazuh Manager VM (Ubuntu Server, hosting the Manager, Indexer, and Dashboard) and the Kali Linux VM, which runs the Wazuh agent and is the only monitored endpoint in this lab setup. Metasploitable 2 was not required for this lab since the focus was on reviewing the Dashboard interface rather than generating a new attack. All VMs remained on the shared Host-Only network, and the Dashboard itself was accessed from a browser on the Windows host over HTTPS, exactly as configured in Part 8 of the guide.

3. Step-by-Step Procedure
I logged into the Wazuh Dashboard from a browser on the Windows host and started on the Overview (Home) screen, the same screen a SOC analyst would check first at the start of a shift.
I reviewed the graph of security events over time along with the breakdown of top alerts, top agents, and severity distribution shown on that home screen.
I then opened the Security Events section and looked through the raw alert list, filtering by time, agent, rule, and severity, and expanded a few individual alerts to see the raw log line behind each one.
Next, I opened the Integrity Monitoring section and reviewed the change history for files that had been added, modified, or deleted on the Kali agent, including the test file from Lab 6.2.
I checked the Vulnerability Detection section to see whether any CVEs had been automatically identified on the Kali agent, noting that this module can take time to fully populate.
Finally, I opened the Security Configuration Assessment (SCA) section and reviewed how the Kali agent's configuration was benchmarked against CIS Benchmark standards, looking at which checks passed and which failed.
For each section, I asked myself what kind of threat it was designed to help detect, what I would do if I saw a high-severity alert there, and where that activity would fit in the NIST Incident Response lifecycle from Lab 4.2.
4. Commands & Actions Performed
Open: Overview (Home)
This is the Dashboard's landing page. It shows a graph of security events over time plus top alerts, top agents, and severity distribution, giving a SOC analyst a single-glance summary of overall system health at the start of a shift.
Open: Security Events
This section lists every raw alert generated by Wazuh's rule engine, filterable by time range, agent, rule, and severity. Clicking an alert expands it to show the exact raw log line that triggered the rule, which is the starting point for investigating any single event.
Open: Integrity Monitoring
This section shows the history of files added, modified, or deleted on monitored agents, including the testfile_wazuh.txt events created earlier in Lab 6.2. Because the change history is retained, this section is also useful for forensic investigations after the fact.
Open: Vulnerability Detection
This section lists CVEs that Wazuh has automatically matched against the software installed on each agent. It functions as a built-in, continuously running vulnerability scanner layered on top of the SIEM, rather than a one-time scan like Nikto in Lab 1.2.
Open: Security Configuration Assessment (SCA)
This section benchmarks an agent's configuration against CIS Benchmark security standards and reports which checks pass or fail. It is Wazuh's automated take on compliance checking, flagging configuration drift away from accepted security baselines.
5. Network / Flow Diagram
  [ SOC Analyst @ Windows Host Browser ]
          |
          | HTTPS (self-signed certificate, port 443)
          v
  [ Wazuh Dashboard ] <---queries---> [ Wazuh Indexer ]  (OpenSearch-based storage)
          ^
          |  alerts/events generated from
          |
  [ Wazuh Manager ] <---logs/events--- [ Kali Linux: Wazuh Agent ]
                                        (auth logs, FIM events, CVE matches, SCA checks)

  Each Dashboard module reads a different slice of that same event stream:
    Overview                   -> aggregated counts and graphs across all event types
    Security Events            -> every individual alert from any matched rule
    Integrity Monitoring       -> FIM events only (file added/modified/deleted, Lab 6.2)
    Vulnerability Detection    -> CVE matches against installed packages
    Security Config Assessment -> CIS benchmark pass/fail results
6. Results & Observations
The Overview screen gave a fast, high-level read on system health, the event graph and severity breakdown made it obvious within seconds whether anything unusual had happened recently, without needing to read individual alerts.
The Security Events section let me trace an alert all the way back to its raw log line, which made it clear how a single log entry on Kali becomes a structured, searchable alert inside the Dashboard.
The Integrity Monitoring section still showed the file added and file modified events from the testfile_wazuh.txt exercise in Lab 6.2, confirming that FIM history persists and is queryable after the fact, not just at the moment an alert fires.
The Vulnerability Detection section took longer to populate with results than the other sections, consistent with the guide's note that this module needs time to complete its scans.
The SCA section showed a mix of passing and failing checks against the CIS Benchmark for the Kali agent, giving a concrete view of where the default configuration diverges from a hardened baseline.
7. What I Learned
This lab reinforced that a SIEM dashboard is organized around the same underlying event stream viewed through different lenses, rather than being a collection of unrelated tools. Overview gives the big picture, Security Events gives the raw detail, Integrity Monitoring and Vulnerability Detection focus on specific categories of risk, and SCA focuses on configuration compliance, but all of it traces back to data the agent is continuously shipping to the Manager.
For SEC+, walking through each section while asking what threat it detects and where it fits in the NIST Incident Response lifecycle from Lab 4.2 helped connect Domain 4's SIEM, SOAR, IOC, and TTP terminology to an actual interface instead of just a list of definitions. It also made clear why SOC analyst roles list SIEM familiarity as a baseline skill: knowing where to look for a given type of evidence is just as important as knowing what the evidence means.
