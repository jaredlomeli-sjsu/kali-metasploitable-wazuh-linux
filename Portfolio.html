This lab made the DMZ concept tangible: high-risk systems belong in their own segment, accessible only through controlled hosts.
LAB 2.2Complete
Secure vs. Insecure Protocols

Compare plaintext Telnet against encrypted SSH to the same target — identical user experience, fundamentally different security properties on the wire.

telnetsshWiresharkKali Linux
View Lab Report
Key Findings
  • Both Telnet and SSH connected to msfadmin@192.168.56.101 — user experience was identical
  • Telnet: credentials visible in plaintext to anyone capturing traffic on the wire
  • SSH: only ciphertext visible — credentials and session data fully encrypted in transit
  • Pattern repeats across the full protocol table: HTTP→HTTPS, FTP→SFTP, SNMP v1/v2→v3, LDAP→LDAPS
Commands Used

Learning Outcome
Two protocols can look and behave identically to users while having completely different security properties underneath. Legacy plaintext protocols should be disabled wherever encrypted alternatives exist — this is one recurring pattern across the entire SEC+ protocol comparison table, not a set of isolated facts to memorize.

Domain 3 — Implementation

Cryptography in practice, firewall configuration, and certificate infrastructure

LAB 3.1Complete
Encryption & Hashing

Demonstrate the avalanche effect in MD5/SHA-256 hashing and perform symmetric AES-256 encryption with GPG to distinguish integrity verification from confidentiality protection.

gpgsha256summd5sumAES-256
View Lab Report
Key Findings
  • Adding a single period to a file produced a completely different MD5 and SHA-256 hash — avalanche effect confirmed
  • MD5: 32-char hex output (128-bit) — SEC+ marks this as broken/weak; SHA-256: 64-char hex (256-bit) — current standard
  • GPG-encrypted file (secret.txt.gpg) was completely unreadable binary without the passphrase
  • Decryption with the correct passphrase recovered the original message exactly — confirmed two-way operation
Commands Used
md5sum testfile.txt sha256sum testfile.txt gpg --symmetric --cipher-algo AES256 secret.txt gpg --decrypt secret.txt.gpg
Learning Outcome
Hashing is one-way and protects integrity. Encryption is two-way and protects confidentiality. MD5 is broken and should not be relied on for security-sensitive checks. SHA-256 is the current standard. AES-256 is the gold standard for symmetric encryption. Doing this by hand made the hashing vs. encryption distinction concrete rather than just definitional.
LAB 3.2Complete
Firewall Rules with iptables

Configure and manage Linux firewall rules using iptables to control inbound and outbound traffic — implementing allow/deny policies and observing their real effect on connectivity.

iptablesKali LinuxMetasploitable
View Lab Report
Key Findings
  • Baseline confirmed: INPUT chain defaulted to ACCEPT — no rules blocking anything before the lab began
  • ICMP DROP rule blocked pings from both Metasploitable and the Windows host — confirmed the rule applies to all sources, not just VMs
  • SSH allow-list demonstrated rule ordering: ACCEPT port 22 must come before the blanket DROP or SSH itself gets blocked
  • Default-deny posture (block all, explicitly allow what's needed) confirmed as far more secure than default-allow
Commands Used
sudo iptables -L -n -v sudo iptables -A INPUT -p icmp --icmp-type echo-request -j DROP sudo iptables -D INPUT -p icmp --icmp-type echo-request -j DROP sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT sudo iptables -A INPUT -p tcp -j DROP sudo iptables -F
Learning Outcome
iptables evaluates rules top-to-bottom and stops at the first match — the same logic behind ACLs on routers and firewalls. The SSH allow-list was a hands-on demonstration of implicit deny: anything not matching an ACCEPT rule above the final DROP is denied by default. This connects directly to stateless vs. stateful firewall concepts and Network Access Control in the SEC+ Implementation domain.
LAB 3.3Complete
Creating & Examining SSL/TLS Certificates

Generate a self-signed X.509 certificate with OpenSSL, inspect every field in plain text, and compare it against a real CA-issued certificate to understand the chain of trust behind HTTPS.

opensslTLS/SSLPKIKali Linux
View Lab Report
Key Findings
  • Self-signed cert: Issuer and Subject fields were identical — I was simultaneously the cert's subject and its own issuer
  • Validity window confirmed: Not Before = creation time, Not After = exactly 365 days later
  • Public Key Algorithm: RSA 2048-bit; Signature Algorithm: SHA-256 — cryptographic signing confirmed, not just plain text
  • Real CA-issued cert (viewed via Firefox padlock) showed Root CA → Intermediate CA → leaf cert chain — this is why browsers trust it without a warning
  • Self-signed certs provide equal encryption strength but carry no third-party validation — hence browser warnings
Commands Used
openssl req -x509 -newkey rsa:2048 -keyout mykey.pem -out mycert.pem -days 365 -nodes openssl x509 -in mycert.pem -text -noout
Learning Outcome
"Encrypted" and "trusted" are not the same thing. A self-signed certificate provides the same RSA encryption as a CA-issued one but carries no third-party validation, which is why browsers reject it by default. This lab made the PKI vocabulary concrete: Root CA, Intermediate CA, CSR, CRL, and OCSP are not isolated terms to memorize — they describe specific roles in the chain of trust that the exam's PKI questions are really testing.

LAB 4.1Complete
Log Analysis

Read auth.log directly, deliberately generate failed logins and a Hydra brute-force attack, then observe how both appear in raw log data — the same signal a SIEM rule keys on.

auth.loggrephydraKali Linux
View Lab Report
Key Findings
  • Every failed login produced a distinct auth.log entry with timestamp, attempted username, and source IP
  • A single manual SSH failure generated a handful of entries; Hydra generated hundreds of "Failed password" lines in seconds from the same source IP
  • The visual difference between baseline noise and a brute-force burst was immediately obvious just by scrolling the log
  • This is exactly the pattern a SIEM correlation rule is built to detect and alert on automatically
Commands Used
sudo cat /var/log/auth.log | head -50 sudo grep "Failed password" /var/log/auth.log ssh wronguser@192.168.56.101 sudo cat /var/log/auth.log | grep "Failed" hydra -l msfadmin -P /usr/share/wordlists/rockyou.txt.gz 192.168.56.101 ssh
Learning Outcome
A single failed login is background noise; a burst of hundreds from one source in seconds is an IOC (Indicator of Compromise) consistent with a brute-force attack. This is the same signal a SIEM platform like Wazuh flags automatically. Reading raw logs by hand first builds the intuition for why log retention, aggregation, and automated correlation rules matter — a human scrolling auth.log does not scale, but the underlying pattern does not change.

Risk Assessment Exercise

Apply the risk formula (Risk = Threat × Vulnerability × Impact) to real Metasploitable vulnerabilities, calculate ALE/SLE/ARO, and practice the four risk response strategies: Accept, Mitigate, Transfer, Avoid.

ALE / SLE / AROrisk matrixGRCnmapcurl
View Lab Report
Key Findings
  • Applied risk formula to two Metasploitable vulnerabilities: open Telnet (port 23) and Apache version disclosure
  • Telnet ALE = $50,000/yr (SLE $100k × ARO 0.5); Apache ALE = $25,000/yr (SLE $100k × ARO 0.25)
  • Telnet carried 2× the annualized risk despite identical High impact and SLE — difference was entirely likelihood and frequency
  • Both resolved to Mitigate: Telnet fix is free (disable, enforce SSH); Apache patch is low-effort, high-return

Learning Outcome
The ALE = SLE × ARO calculation became concrete by applying it to vulnerabilities I had personally found rather than textbook numbers. The right risk response depends on fix cost relative to the ALE it removes — in both cases here, Mitigate was obvious because the fix cost was near zero compared to the dollar risk eliminated. Also reinforced the asset/threat/vulnerability/risk distinction the exam tests.

Tools & Technologies
Security tools, platforms, and protocols covered across all labs
🔍
Nmap
Network discovery, port scanning, service & OS fingerprinting
Definition
Open-source network scanner that sends crafted packets and analyzes responses to map live hosts, open ports, running services and their versions, and the underlying operating system — all without needing credentials on the target.
Use Case in This Lab
Used in Lab 1.1 to sweep the 192.168.56.0/24 subnet, confirm Metasploitable at .101, then fingerprint every open port and service. The output became the attack surface map used in all later labs.
⚡
Nikto
Web server vulnerability scanner, OSVDB/CVE mapping
Definition
Web server scanner that checks for thousands of known vulnerabilities, outdated software versions, dangerous files, and missing security headers. Results are mapped to OSVDB and CVE identifiers for reference. It enumerates — it does not exploit.
Use Case in This Lab
Used in Lab 1.2 against Metasploitable's Apache server. Returned dozens of findings including outdated Apache version, missing X-Frame-Options headers, and exposed directories — all tagged with OSVDB IDs.
🔑
Hydra
Automated credential brute-force / dictionary attack
Definition
Fast, parallelized login cracker that automates dictionary and brute-force attacks against network services (FTP, SSH, HTTP, Telnet, and many more). Tries username/password combinations from a wordlist at high speed until one succeeds.
Use Case in This Lab
Used in Lab 1.3 to crack msfadmin:msfadmin on FTP in seconds using the rockyou wordlist — no lockout policies were in place, so it ran unchecked until it found the match.
📡
Wireshark
Packet capture and protocol analysis
Definition
Network protocol analyzer that captures live traffic from a network interface and lets you inspect individual packets layer by layer — from Ethernet frames down to application-layer payloads. Display filters isolate specific protocols or conversations.
Use Case in This Lab
Used in Lab 4.3 to capture traffic on eth0. The telnet filter revealed plaintext credentials in packet payloads; the ssh filter showed only encrypted binary — confirming why protocol choice determines whether data is exposed.
🔐
GPG / AES-256
Symmetric encryption and key management
Definition
GNU Privacy Guard (GPG) implements OpenPGP standards for encryption and signing. AES-256 (Advanced Encryption Standard, 256-bit key) is the symmetric cipher used for bulk data — it is the current gold standard for confidentiality, used in TLS, disk encryption, and file protection.
Use Case in This Lab
Used in Lab 3.1 to encrypt a secret message with gpg --symmetric --cipher-algo AES256, confirm the output was unreadable binary, then decrypt it with the correct passphrase — demonstrating two-way confidentiality protection.
🛡️
iptables
Linux netfilter firewall rule configuration
Definition
Linux kernel's built-in packet filtering tool, part of the Netfilter framework. Rules in the INPUT, OUTPUT, and FORWARD chains are evaluated top-to-bottom; the first matching rule wins. Supports filtering by protocol, port, source/destination IP, and connection state.
Use Case in This Lab
Used in Lab 3.2 to block ICMP pings from all sources, then build an SSH-only allow-list — demonstrating implicit deny: everything not explicitly accepted by an earlier rule falls through to the final DROP. Rule ordering was confirmed to matter by swapping the sequence.
📜
OpenSSL
Certificate generation, TLS inspection, crypto operations
Definition
Open-source cryptography toolkit implementing SSL/TLS protocols and a wide range of cryptographic primitives. Used to generate key pairs, create and sign certificates, inspect certificate fields, test TLS handshakes, and perform hashing and encryption from the command line.
Use Case in This Lab
Used in Lab 3.3 to generate a self-signed RSA 2048-bit certificate (req -x509) and inspect every field with x509 -text -noout — Issuer, Subject, validity window, and signature algorithm — making PKI structure visible.
🖥️
Kali Linux
Attacker / analyst VM — all tools run from here
Definition
Debian-based Linux distribution maintained by Offensive Security and purpose-built for penetration testing and security auditing. Ships pre-installed with hundreds of tools covering reconnaissance, exploitation, post-exploitation, forensics, and wireless security.
Use Case in This Lab
Served as the attacker and analyst workstation for every lab in this series. All scanning, cracking, packet capture, hashing, encryption, firewall configuration, and certificate generation commands ran from Kali's terminal against Metasploitable as the target.
🎯
Metasploitable 2
Intentionally vulnerable target VM
Definition
Intentionally insecure Ubuntu-based virtual machine created by Rapid7 for practicing exploitation in a safe, legal environment. It runs deliberately outdated and misconfigured services — Telnet, FTP, vulnerable Apache, weak credentials — to provide realistic attack surfaces without risk to production systems.
Use Case in This Lab
Served as the target for every domain in this lab series. Its open ports (21, 22, 23, 80, etc.), weak default credentials (msfadmin:msfadmin), outdated Apache, and insecure services were the real vulnerabilities scanned, cracked, and analyzed across labs 1.1 through 4.3.
📊
Wazuh SIEM
Event monitoring, FIM, alert correlation — SIEM detection labs
Definition
Free, open-source Security Information and Event Management (SIEM) and XDR platform used in real SOC environments worldwide. Collects logs from agents on monitored endpoints, applies detection rules, correlates events into alerts, performs File Integrity Monitoring (FIM), and provides a searchable dashboard for threat hunting.
Use Case in This Lab
Used in Lab 6.1 to detect the same Hydra brute-force attack from Lab 1.3 from the defender's perspective — Security Events spike, Threat Hunting filters, and correlation alerts. Labs 6.2–6.3 (FIM and dashboard exploration) are upcoming.
💻
VirtualBox
Virtualization host; Host-Only network isolation
Definition
Free, open-source Type 2 hypervisor from Oracle that runs multiple operating systems as virtual machines on a single physical host. Supports multiple network modes: Host-Only (VMs talk to each other, not the internet), NAT (VM gets internet via host), and Bridged (VM appears as a real network device).
Use Case in This Lab
Hosts all VMs (Kali, Metasploitable, future Wazuh) and provides the Host-Only adapter that creates the isolated 192.168.56.0/24 lab network. Lab 2.1 directly demonstrated segmentation by adding a NAT adapter to Kali while keeping Metasploitable Host-Only only.
📖
rockyou.txt
Common-password wordlist for brute-force demonstrations
Definition
A 14-million-entry plaintext password list leaked from the 2009 RockYou data breach, where passwords were stored unencrypted. It is now the most widely used wordlist in penetration testing because it reflects real passwords real people chose — common words, names, and patterns that weak password policies allow.
Use Case in This Lab
Supplied to Hydra in Lab 1.3 as the dictionary for the FTP brute-force attack. The password msfadmin appeared early in the list because it matches common patterns (username = password), demonstrating why predictable credentials are cracked in seconds.
